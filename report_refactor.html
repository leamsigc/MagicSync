MagicSync Refactor Playbook

Step-by-step, TDD-guided deepening of five architectural seams.

Intended for: smaller models working incrementally. Do one candidate at a time.

Candidates

  1. Scheduler Plugin Interface
  2. BulkSchedulerService
  3. PlatformStatsService
  4. AI Tools Facade
  5. Asset Service Duplication

How to Use This Playbook

  • Work on one candidate at a time.
  • Follow the must-do points in order.
  • Use the TDD loop for every new module: write a failing test, make it pass, refactor.
  • Run the validation commands after each candidate.
  • Never modify more than one candidate before running tests.

Scheduler Plugin Interface

Current Problem

The SchedulerPlugin interface in packages/scheduler/server/services/SchedulerPost.service.ts forces every platform adapter to implement validate, post, update, addComment, getStatistic, getComments, and replyToComment. This is a shallow, broad interface.

Goal

Split the monolithic interface into capability-specific seams: PostPublisher, StatsProvider, and CommentManager. A plugin registers only the capabilities it supports.

Must-Do Points

  1. Read packages/scheduler/server/services/SchedulerPost.service.ts and list every method currently required by SchedulerPlugin.
  2. Group methods by capability: posting, statistics, comments.
  3. Define three new interfaces in packages/scheduler/server/types/scheduler-plugin.types.ts:
    • PostPublisher: validate, post, update
    • StatsProvider: getStatistic
    • CommentManager: getComments, replyToComment, addComment
  4. Replace the single SchedulerPlugin interface with a registry that maps platform names to capability sets.
  5. Update every plugin in packages/scheduler/server/services/plugins/ to implement only the capabilities it supports.
  6. Update AutoPost.service.ts and SchedulerPost.service.ts to look up capabilities before calling methods.
  7. Remove any stub methods that plugins implemented only to satisfy the old interface.

TDD Loop

  1. Red: Write a test that creates a plugin implementing only PostPublisher and asserts it can be registered without implementing comment or stats methods.
  2. Green: Update the registry to accept capability-specific interfaces and make the test pass.
  3. Refactor: Extract helper functions for capability lookup so the registry code is readable.
  4. Red: Write a test that calls getStatistic on a plugin that does not support it and expects a clear error.
  5. Green: Implement capability checking in SchedulerPost.service.ts.
  6. Refactor: Ensure every capability check uses the same helper.

Files to Create/Modify

  • Create: packages/scheduler/server/types/scheduler-plugin.types.ts
  • Modify: packages/scheduler/server/services/SchedulerPost.service.ts
  • Modify: packages/scheduler/server/services/AutoPost.service.ts
  • Modify: packages/scheduler/server/services/plugins/*.plugin.ts
  • Tests: packages/scheduler/server/services/SchedulerPost.service.test.ts

Validation

pnpm build
pnpm test:scheduler
# or if no test script exists:
cd packages/scheduler && pnpm test

BulkSchedulerService

Current Problem

BulkSchedulerService in packages/bulk-scheduler/server/services/bulkScheduler.service.ts handles CSV import, bulk generation, template processing, date distribution, asset downloading, platform trimming, and notifications. It is a shallow god-object.

Goal

Introduce deep modules behind narrow seams: CsvPostImporter, BulkPostGenerator, PlatformContentFormatter, RemoteAssetDownloader, and BulkPostBatcher.

Must-Do Points

  1. Read packages/bulk-scheduler/server/services/bulkScheduler.service.ts and identify every distinct responsibility.
  2. Extract trimContentForPlatform into packages/scheduler/server/utils/platformFormatter.ts in the BaseScheduler layer.
  3. Extract downloadAndCreateAsset into packages/assets/server/utils/remoteDownloader.ts in the BaseAssets layer.
  4. Create CsvPostImporter in packages/bulk-scheduler/server/services/csvImporter.service.ts with a single public method import(csvBuffer, options).
  5. Create BulkPostGenerator in packages/bulk-scheduler/server/services/bulkGenerator.service.ts with a single public method generate(template, variables, options).
  6. Create BulkPostBatcher in packages/bulk-scheduler/server/services/bulkPostBatcher.service.ts that wraps postService.create with Promise.allSettled.
  7. Refactor BulkSchedulerService to accept the new modules via constructor and delegate all work to them.
  8. Ensure no inline CSV parsing, template expansion, or asset download logic remains in BulkSchedulerService.

TDD Loop

  1. Red: Write a test for PlatformContentFormatter.trim with a string longer than the platform limit; expect it to truncate correctly.
  2. Green: Move the trimming logic into the new formatter and make the test pass.
  3. Refactor: Remove the old inline method from BulkSchedulerService.
  4. Red: Write a test for CsvPostImporter.import that passes a CSV buffer and expects an list of post objects.
  5. Green: Move CSV parsing logic into the importer and make the test pass.
  6. Refactor: Update BulkSchedulerService to use the importer.
  7. Red: Write a test for BulkPostBatcher that mocks postService.create and asserts all promises are settled.
  8. Green: Implement the batcher and wire it into the service.

Files to Create/Modify

  • Create: packages/bulk-scheduler/server/services/csvImporter.service.ts
  • Create: packages/bulk-scheduler/server/services/bulkGenerator.service.ts
  • Create: packages/bulk-scheduler/server/services/bulkPostBatcher.service.ts
  • Create: packages/scheduler/server/utils/platformFormatter.ts
  • Create: packages/assets/server/utils/remoteDownloader.ts
  • Modify: packages/bulk-scheduler/server/services/bulkScheduler.service.ts

Validation

cd packages/bulk-scheduler && pnpm test

PlatformStatsService

Current Problem

PlatformStatsService fetches stats, persists snapshots, calculates growth, aggregates, and builds time-series. It mixes integration, persistence, and presentation concerns.

Goal

Split into StatsCollector, StatsRepository, and StatsAggregator. The service keeps only orchestration.

Must-Do Points

  1. Read packages/scheduler/server/services/PlatformStats.service.ts and classify each method as collect, persist, or aggregate.
  2. Create StatsRepository in packages/db/server/services/stats.repository.ts with methods: getLatestSnapshot, saveSnapshot, getStatsHistory.
  3. Create StatsCollector in packages/scheduler/server/services/StatsCollector.service.ts with methods: fetchAccountStats, collectAllStats.
  4. Create StatsAggregator in packages/scheduler/server/services/StatsAggregator.service.ts with methods: calculateGrowth, getAggregatedByPlatform, getTimeSeriesData.
  5. Refactor PlatformStatsService to depend on the three new modules via constructor.
  6. Move all Drizzle queries out of PlatformStatsService and into StatsRepository.
  7. Update API routes in packages/scheduler/server/api/v1/stats/ to use the new modules directly where appropriate.

TDD Loop

  1. Red: Write a test for StatsRepository.saveSnapshot using an in-memory Drizzle mock; expect the correct insert/update behavior.
  2. Green: Move persistence logic into the repository and make the test pass.
  3. Red: Write a test for StatsAggregator.calculateGrowth with two snapshots; expect the correct percentage change.
  4. Green: Move growth calculation into the aggregator and make the test pass.
  5. Red: Write a test for StatsCollector.collectAllStats that mocks the external platform call and asserts the repository receives the data.
  6. Green: Move collection logic into the collector and make the test pass.
  7. Refactor: Reduce PlatformStatsService to a thin orchestrator.

Files to Create/Modify

  • Create: packages/db/server/services/stats.repository.ts
  • Create: packages/scheduler/server/services/StatsCollector.service.ts
  • Create: packages/scheduler/server/services/StatsAggregator.service.ts
  • Modify: packages/scheduler/server/services/PlatformStats.service.ts
  • Modify: packages/scheduler/server/api/v1/stats/*.ts

AI Tools Facade

Current Problem

AiToolsFacadeService in packages/ai-tools/server/services/aiToolsFacade.service.ts handles authentication, LLM context, thread management, document handling, and skill management. A single facade with such a broad surface is shallow.

Goal

Replace the monolithic facade with deep modules: ChatSessionManager, SkillRegistry, DocumentContext, and LLMGateway.

Must-Do Points

  1. Read packages/ai-tools/server/services/aiToolsFacade.service.ts and map every public method to one of: chat, skills, documents, LLM gateway.
  2. Create LLMGateway in packages/ai-tools/server/services/llmGateway.service.ts to own all Python-backend HTTP calls, JWT context, and SSE parsing.
  3. Create ChatSessionManager in packages/ai-tools/server/services/chatSessionManager.service.ts to own thread lifecycle and message history.
  4. Create SkillRegistry in packages/ai-tools/server/services/skillRegistry.service.ts to own skill loading, matching, and invocation.
  5. Create DocumentContext in packages/ai-tools/server/services/documentContext.service.ts to own document upload, chunking, and retrieval.
  6. Delete or deprecate AiToolsFacadeService after all callers are updated.
  7. Update API routes in packages/ai-tools/server/api/ai-tools/ to import the specific module they need.

TDD Loop

  1. Red: Write a test for LLMGateway.sendChat that mocks the Python backend and asserts the correct JWT is attached.
  2. Green: Move the HTTP/SSE logic from the facade into the gateway and make the test pass.
  3. Red: Write a test for ChatSessionManager.createThread that asserts a thread record is created.
  4. Green: Move thread logic into the manager and make the test pass.
  5. Red: Write a test for SkillRegistry.invoke with a known skill name; expect the correct handler to run.
  6. Green: Move skill logic into the registry and make the test pass.
  7. Refactor: Update chat endpoint to compose the four modules instead of using the facade.

Files to Create/Modify

Asset Service Duplication

Current Problem

Asset-related logic appears in packages/assets, packages/db, and packages/shared. The seam is not real because there is no single interface callers can rely on.

Goal

Define a single IAssetService interface in BaseShared, move the Drizzle implementation into BaseDB, and make BaseAssets own domain orchestration.

Must-Do Points

  1. Read the three asset service files and identify which one contains the real implementation.
  2. Define IAssetService in packages/shared/server/types/asset.types.ts with methods like createAsset, getAssetById, deleteAsset.
  3. Move the concrete Drizzle implementation into packages/db/server/services/asset.service.ts and make it implement IAssetService.
  4. Delete the re-export files in packages/assets/server/services/asset.service.ts and packages/shared/server/services/asset.service.ts.
  5. Update packages/assets/server/utils/AssetsUtils.ts to import the DB implementation from #layers/BaseDB/server/services/asset.service.
  6. Search the entire monorepo for imports of the old asset service paths and update them.
  7. Ensure no layer other than BaseDB directly imports Drizzle schema for assets.

TDD Loop

  1. Red: Write a test that asserts packages/db/server/services/asset.service.ts implements IAssetService (type-level or runtime check).
  2. Green: Move the implementation and make the test pass.
  3. Red: Write a test for createAsset using an in-memory DB mock.
  4. Green: Ensure the DB service passes the test.
  5. Red: Write a test that imports the asset service from the correct layer and asserts it is defined.
  6. Green: Fix all broken imports across the monorepo.

Files to Create/Modify

  • Create: packages/shared/server/types/asset.types.ts
  • Modify: packages/db/server/services/asset.service.ts
  • Delete: packages/assets/server/services/asset.service.ts
  • Delete: packages/shared/server/services/asset.service.ts
  • Modify: packages/assets/server/utils/AssetsUtils.ts

Global Rules

  • Never modify more than one candidate before running tests.
  • Always run pnpm build after file moves or import changes.
  • Prefer constructor injection over direct imports for new modules.
  • Keep Nuxt layer boundaries: BaseShared for types, BaseDB for persistence, BaseAssets for file logic, BaseScheduler for scheduling rules.
  • Record any new ADR if a refactor contradicts an existing decision.
