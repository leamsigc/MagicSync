## CRITICAL — Fix Immediately

### SECURITY

#### C1: OAuth Tokens + Full Account Objects Written to Audit Logs
- **File:** `packages/auth/lib/auth.ts` — lines 374, 401, 426, 431, 462, 494, 547, 560
- **Severity:** CRITICAL
- **Issue:** Audit log entries include raw `access_token`, `refresh_token`, and the full account object in the `details` field:
  ```ts
  // Line 547 — full account including tokens
  details: '${JSON.stringify(account)} from ${account.providerId}'
  // Line 560 — same, plus error details
  details: '${JSON.stringify(account)} from ${account.providerId}, Error ${error}'
  ```
- **Impact:** Any log aggregation system (S3, Datadog, Elasticsearch) will contain raw OAuth tokens. These can be used to impersonate users on connected platforms.
- **Fix:** Redact all token fields before logging. Never log `accessToken`, `refreshToken`, `accountId`, or full OAuth response objects.

#### C2: Account Unlink Has No Ownership Check — IDOR Vulnerability
- **File:** `packages/auth/server/services/account.service.ts:30-32`
- **Severity:** CRITICAL
- **Issue:**
  ```ts
  const result = await db.delete(account)
    .where(eq(account.id, accountId))   // ← NO userId check
    .returning()
  ```
  Any authenticated user who knows another user's `accountId` can delete their social account linkage. The `userId` is passed to the service but never used in the WHERE clause.
- **Fix:** Add `eq(account.userId, currentUser.id)` to the WHERE clause.

#### C3: Insecure TrustedOrigins Fallback
- **File:** `packages/auth/lib/auth.ts:23-26`
- **Severity:** CRITICAL
- **Issue:**
  ```ts
  trustedOrigins: [
    process.env.NUXT_BETTER_AUTH_URL || 'http://localhost:3000',
    "127.0.0.1:3000"   // ← missing protocol, non-TLS
  ]
  ```
  If `NUXT_BETTER_AUTH_URL` is unset in production, the entire auth cookie layer accepts `http://127.0.0.1:3000` as a valid origin.
- **Fix:** Remove the insecure fallback. Fail at startup if the env var is not set in non-dev environments.

#### C4: `BETTER_AUTH_SECRET` Crashes Server on Missing Env
- **Files:** `packages/auth/server/utils/AuthHelpers.ts:68, 78`
- **Severity:** CRITICAL
- **Issue:**
  ```ts
  key: process.env.BETTER_AUTH_SECRET!   // ← throws if missing
  ```
  The server hard-crashes if `BETTER_AUTH_SECRET` is absent. No startup validation.
- **Fix:** Add startup validation. Fail fast with a clear message if required env vars are missing.

#### C5: Authorization Header + Cookies Logged on Every Error
- **File:** `packages/auth/server/middleware/errorHandler.ts:189-191`
- **Severity:** CRITICAL
- **Issue:**
  ```ts
  context: {
    headers: event.node.req.headers,  // ← includes Authorization, cookies
  }
  ```
  When any unhandled error occurs, all request headers (including `Authorization`, cookies, `x-api-key`) are embedded in the log context.
- **Fix:** Explicitly whitelist safe headers. Never log `authorization`, `cookie`, `set-cookie`.

#### C6: Full User Object Logged at INFO on Every Authenticated Request
- **File:** `packages/auth/server/middleware/001.auth.ts:12`
- **Severity:** CRITICAL
- **Issue:**
  ```ts
  log.info({ user })   // ← logs ALL fields including email, id, role
  ```
- **Fix:** Only log safe user fields (e.g., `user.id`). Never log email, full role, or image URLs at INFO level on every request.

### ARCHITECTURE

#### C10: API Key Auth Blocked by Session-Only Global Middleware

#### C11: `publicApiPrefixes` Allowlist Is a Maintenance Trap

### DATABASE

#### C12: Missing Transactions — `post.service.ts` Create & Update
- **File:** `packages/db/server/services/post.service.ts:33-101` (`create`) and `:254-326` (`update`)
- **Severity:** HIGH
- **Issue:** Both methods perform multiple DB writes (insert post, update assets, insert/delete platform posts) without a `db.transaction()`. If a step fails midway, the DB is left in a partial state.
- **Fix:** Wrap all related writes in `db.transaction()`.

#### C13: Missing Transactions — Social Media Account Create/Update
- **File:** `packages/db/server/services/social-media-account.service.ts:269-322`
- **Severity:** HIGH
- **Issue:** `createOrUpdateAccount` performs get + update/create + entity details operations without a transaction.
- **Fix:** Wrap in `db.transaction()`.

#### C14: Service Methods Throwing Instead of Returning `ServiceResponse`
- **Files:**
  - `packages/db/server/services/post.service.ts:244` — `findByIdFull` throws `Error`
  - `packages/db/server/services/social-media-account.service.ts:135` — `createAccount` throws `Error`
  - `packages/db/server/services/entity-details.service.ts:45` — `createDetails` throws `Error`
- **Severity:** HIGH
- **Issue:** Inconsistent with the established `ServiceResponse<T>` pattern. Thrown errors don't carry structured error codes, and callers can't handle them gracefully.
- **Fix:** Replace all `throw new Error(...)` with `return { error: '...' }` in service methods.

#### C15: N+1 Queries in `post.service.ts`
- **Files:**
  - `server/services/post.service.ts:180-192` — `findByBusinessId` loops over posts calling `this.db.query.assets.findMany`
  - `server/services/post.service.ts:69-92` — `create` calls `getAccountById` in a loop for each target platform
  - `server/services/social-media-account.service.ts:97-113` — `getUserAccountsCompleteDetails` loops over entity details
- **Severity:** HIGH
- **Fix:** Batch queries using `Promise.all` with single queries, or use Drizzle relations with `.with()` to prefetch.

#### C16: Missing Database Indexes on Frequently Queried Columns
- **Files:**
  - `db/posts/posts.ts:9-26` — posts table: no indexes on `userId`, `businessId`, `status`, `scheduledAt`
  - `db/rag/rag.ts:18-36` — documents: no indexes on `userId`, `folderId`, `status`
  - `db/rag/rag.ts:39-50` — documentChunks: no index on `documentId`
  - `db/business/business.ts` — businessProfiles: no composite index on `(userId, isActive)`
  - `db/socialMedia/socialMedia.ts` — socialMediaAccounts: no composite index on `(userId, businessId, isActive)`
- **Severity:** MEDIUM
- **Fix:** Add indexes for all WHERE-clause filter columns. For posts: `(userId, status, scheduledAt)`, `(businessId, status)`. For socialMediaAccounts: `(userId, isActive)`.

## HIGH Priority

- **File:** `packages/auth/app/middleware/02.admin.global.ts:5`
- **Issue:**
  ```ts
  const isUserNavigatingToAdminOnly = to.path.startsWith('admin')  // 'admin' not '/admin'
  ```
  Matches nothing. Admin routes are **not protected** by this middleware.
- **Fix:** Change to `to.path.startsWith('/admin')` or `to.path.startsWith('/app/admin')`.

#### H2: CLI Endpoint Passes Empty `userId` — Ownership Bypass
- **File:** `packages/scheduler/server/api/v1/cli/post.post.ts:139`
- **Issue:**
  ```ts

  ```
  Passes empty string as `userId`, likely bypassing ownership checks.
- **Fix:** Extract the actual user from the API key's associated account and pass the correct `userId`.

#### H3: Unbounded Rate Limiting Gap
- **Files:** All packages under `server/api/`
- **Issue:** No rate limiting middleware on any endpoint. Schema has rate limit fields but they are not enforced. All public API endpoints are vulnerable to abuse.
- **Fix:** Implement `@unjs/ratelimit` or similar on all public endpoints.

#### H4: `getFullOrganization` Leaks Org Membership Without Auth Check
- **File:** `packages/auth/server/utils/useAuthApi.ts:59-64`
- **Issue:** better-auth's `getFullOrganization` returns org details including member list to any authenticated caller. Callers like `business-org.service.ts` rely on post-hoc membership checks.
- **Fix:** Ensure callers always verify membership. Consider wrapping `getFullOrganization` to enforce membership check internally.

#### H5: Missing Input Validation — Multiple Endpoints
- **Files:**
  - `packages/scheduler/server/api/v1/posts/index.post.ts:13` — raw `readBody()` without Zod
  - `packages/scheduler/server/api/v1/posts/index.put.ts:29` — raw `readBody()` without Zod
  - `packages/connect/server/api/v1/social-accounts/api-key/[platform].post.ts:13` — raw `readBody()`
  - `packages/ai-tools/server/api/ai-tools/chat.post.ts:10` — raw `readBody()`
  - `packages/scheduler/server/api/v1/cli/post.post.ts:126` — raw `readBody()`
  - `packages/bulk-scheduler/server/api/v1/bulk-scheduler/csv-import.post.ts:32` — `JSON.parse` on user input without try-catch
- **Fix:** Use `readValidatedBody()` with Zod schemas for all endpoints. Wrap `JSON.parse` in try/catch.

#### H6: Unsafe JSON Parse in API Key Service
- **File:** `packages/auth/server/services/api-key.service.ts:84`
- **Issue:**
  ```ts
  const metadata = keyData.metadata ? JSON.parse(keyData.metadata as any || '{}') : {}
  ```
  If `metadata` is malformed, `JSON.parse` throws unhandled, crashing the request.
- **Fix:** Wrap in try/catch with a fallback.

#### H7: Frontend — `ref()` Instead of `useState()` for SSR-Safe State
- **Files:**
  - `packages/site/app/composables/usePlatformStats.ts:54-59`
  - `packages/site/app/composables/useDashboardMetrics.ts:30-43`
  - `packages/auth/app/composables/useApiKeyManagement.ts:28-32`
- **Issue:** Using `ref()` for shared state in SSR apps causes state loss across hydration boundaries. Should use `useState()`.
- **Fix:** Replace `ref()` with `useState()` for composable state that needs SSR persistence.

#### H8: Frontend — No Typed API Client Layer
- **Files:**
  - `packages/site/app/composables/usePlatformStats.ts` — direct `$fetch()` calls
  - `packages/auth/app/composables/useApiKeyManagement.ts` — direct `$fetch()` calls
  - `packages/auth/app/composables/useNotificationManagement.ts` — multiple `$fetch()` calls
- **Issue:** No type validation of API responses at compile time. Type drift between frontend and backend types.
- **Fix:** Create a shared types package and a typed API client wrapper that all composables use.

#### H9: Frontend — `console.error` in Production Code
- **Files:**
  - `packages/site/app/composables/useDashboardMetrics.ts:99-100` — `console.error('Error fetching dashboard metrics:', err)`
  - `packages/site/app/composables/usePlatformStats.ts` — multiple `console.error` calls
- **Issue:** `console.error` leaks to production logs and is not filterable/structured.
- **Fix:** Use a proper logging framework (e.g., Nuxt's `useLogger()`) or at minimum use `useState()` error state to trigger UI feedback without logging.

#### H10: `unread-count.get.ts` Selects All Rows Just to Count
- **File:** `packages/auth/server/api/v1/notifications/unread-count.get.ts`
- **Issue:**
  ```ts
  const result = await db.select().from(notifications).where(...)
  return { count: result.length }   // ← loads ALL rows into memory
  ```
- **Fix:** Use `db.select({ count: count() }).from(notifications)` (Drizzle aggregate).
