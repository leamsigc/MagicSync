Social Media Account Service

packages/db/server/services/social-media-account.service.ts

Strong

Files Involved

  • packages/db/server/services/social-media-account.service.ts
  • packages/db/server/services/oauth.service.ts (new)
  • packages/db/server/services/token.service.ts (new)
  • packages/db/server/services/account.service.ts (new)

Problem

Interface vs Implementation Complexity: The service has 25+ methods covering creation, updates, deletion, token management, validation, and account linking, yet the implementation is a 747-line single class with many thin wrappers.
Tight Coupling Across Seams: Direct imports from BaseDB schema and BaseAuth, uses useDrizzle(), and calls entityDetailsService from another service.
Lack of Locality: Pure functions like isTokenExpired() and getTokenHealth() are extracted for testability but called from multiple places with different types.
Untested/Hard to Test Areas: OAuth token refresh logic, account validation against external APIs, token expiration checks with timezone considerations.

Solution

Refactor into smaller, more focused services: Split into AccountService (CRUD operations), TokenService (token management), and OAuthService (external API interactions).
Use dependency injection: Make the service more testable by injecting dependencies instead of direct imports.

Benefits

  • Locality: Each service has a clear, focused responsibility
  • Leverage: Changes to one service will have less impact on others
  • Testability: Smaller services are easier to test
  • Maintainability: The system will be easier to understand and modify

Extract common plugin logic: Move shared logic into a base class or utility functions.
Move plugin discovery: Change from hardcoded object to configuration file.
Add testing harness: Create comprehensive tests for plugin validation.

Benefits

  • Locality: Plugin discovery is centralized and easy to understand
  • Leverage: Adding new plugins becomes straightforward
  • Testability: Plugins can be tested independently
  • Maintainability: Clear separation between plugin logic and scheduler

CRITICAL: Direct Database Queries in API Routes

Problem

Violation of Non-Negotiables: Admin API routes in `packages/site/server/api/v1/admin/` are using direct database queries instead of the service layer pattern.
Specific Violations:
  • Direct `useDrizzle()` calls in route handlers
  • Direct `db.select().all()` queries
  • Missing `ServiceResponse` return types

Impact

  • Architecture Violation: Breaks the established service layer pattern
  • Testing Difficulty: Harder to test route handlers without duplicating business logic
  • Maintainability: Changes to business logic require updates in multiple places
  • Security: Missing proper error handling and validation

Solution

Refactor to use service layer: Create or extend existing services to handle admin operations, then have API routes call these services.
Consistent Pattern: Ensure all API routes follow the same pattern as other services.

Recommendation Strength

⭐⭐⭐⭐⭐ (Critical)

Cross-Cutting Architectural Issues

1. Service Layer Over-Engineering

The service layer is becoming increasingly complex, with many services that have overlapping responsibilities. Understanding any business operation requires reading through multiple service files.

Solution: Consolidate related services into fewer, more focused modules.

2. Database Coupling

Services directly import from BaseDB schema and use useDrizzle(). Tight coupling to the database schema makes refactoring difficult.

Solution: Use a more abstract repository pattern that decouples services from the schema.

3. Lack of Clear Boundaries

Services often handle multiple concerns (CRUD, validation, business logic, external API calls). Services become large and complex, making them hard to maintain.

Solution: Apply strict single-responsibility principle to services.

4. Testing Gaps

Many services are not well tested, especially those with complex business logic. Bugs are hard to detect and fix.

Solution: Add comprehensive unit tests for all service methods, especially those with pure functions.
